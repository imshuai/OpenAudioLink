# Phase 1-A Network Handshake Design

Status: Ready for user review
Date: 2026-07-08

## Goal

Phase 1-A proves the Version 1 protocol can run over TCP between the Android Sender and Windows Receiver before adding audio capture, AAC decoding, discovery, or installer work.

The deliverable is a small network handshake skeleton:

```text
TCP connect
HELLO -> WELCOME
START_STREAM -> STREAM_READY
PING -> PONG
STOP_STREAM -> close
```

## Scope

Included:

- Protocol serialization for existing parsed packet structures.
- Payload builders/parsers for `HELLO`, `WELCOME`, `START_STREAM`, `STREAM_READY`, `PING`, `PONG`, `STOP_STREAM`, and minimal `ERROR`.
- Windows TCP listener that accepts one active sender.
- Android TCP client that connects to an explicit host/port.
- Session state transitions for handshake, ready, streaming, stopping, and disconnected.
- Unit tests for serialization and state transitions.
- One loopback integration path using fake peers where the local platform can run it.

Excluded:

- MediaProjection and AudioPlaybackCapture.
- AAC encode/decode and PCM playback.
- mDNS discovery.
- Foreground service.
- Windows installer.
- Full UI beyond a minimal start/stop hook if needed to launch the skeleton manually.

## Protocol Decisions

Phase 1-A follows `docs/03-Protocol.md` exactly for packet framing:

- Header size: `24` bytes.
- Byte order: Big Endian.
- Magic: `OALP`.
- Protocol version: `1.0`.
- Maximum payload size remains bounded by `ProtocolConstants.MaxPacketSize`.

Packet types used in Phase 1-A:

| Type | Name |
|-----:|------|
| `0x01` | `HELLO` |
| `0x02` | `WELCOME` |
| `0x03` | `START_STREAM` |
| `0x04` | `STREAM_READY` |
| `0x06` | `STOP_STREAM` |
| `0x07` | `PING` |
| `0x08` | `PONG` |
| `0x09` | `ERROR` |

`AUDIO` remains parsed and validated from Phase 0 but is not sent in Phase 1-A.

## Message Payloads

`HELLO` payload:

```text
Sender Name      UTF-8 String
Sender Version   UTF-8 String
Protocol Major   UInt8
Protocol Minor   UInt8
Platform         UInt8
Capabilities     UInt32
```

`WELCOME` payload:

```text
Result            UInt8
Receiver Name     UTF-8 String
Receiver Version  UTF-8 String
Session ID        UInt64
```

`START_STREAM` payload:

```text
Codec             UInt8
Sample Rate       UInt32
Channels          UInt8
Bitrate           UInt32
Frame Duration    UInt16
```

Phase 1-A sends the canonical Version 1 request:

```text
Codec = AAC-LC
Sample Rate = 48000
Channels = 2
Bitrate = 192000
Frame Duration = 20
```

`STREAM_READY` payload:

```text
Result                  UInt8
Negotiated Codec        UInt8
Negotiated Sample Rate  UInt32
Negotiated Channels     UInt8
```

`PING` and `PONG` payload:

```text
Sequence    UInt32
Timestamp   UInt64
```

`PONG` payload must be byte-for-byte identical to the received `PING` payload.

`STOP_STREAM` has an empty payload.

`ERROR` payload:

```text
Error Code   UInt16
Severity     UInt8
Description  UTF-8 String
```

## Component Design

### Shared Protocol Shape

Android and Windows each get the same small set of protocol responsibilities:

- `ProtocolConstants`: packet type constants and fixed protocol values.
- `PacketParser`: existing header and payload extraction.
- `PacketWriter` or equivalent minimal serializer: writes header and payload bytes.
- Message-specific payload helpers for strings and fixed-width integers.

No shared cross-language generator is added in Phase 1-A. The existing Python golden packet generator remains the source for common binary fixtures.

### Windows Receiver

Windows adds a receiver-side network skeleton:

```text
TcpListener
  -> accept TcpClient
  -> read one framed packet at a time
  -> SessionManager handles packet order
  -> write response packet
```

The receiver accepts one active session. A second sender receives `WELCOME(Result = Receiver Busy)` and the connection closes.

Session IDs are `UInt64` values generated by the receiver. A monotonic in-process counter is enough for Phase 1-A because uniqueness is only required until receiver restart.

### Android Sender

Android adds a sender-side client skeleton:

```text
Socket
  -> send HELLO
  -> wait WELCOME
  -> send START_STREAM
  -> wait STREAM_READY
  -> send PING
  -> wait PONG
  -> send STOP_STREAM
```

The first UI path may use explicit host and port constants or temporary fields. Discovery is intentionally not part of Phase 1-A.

## State Machines

Sender states:

```text
Idle -> Connecting -> Handshake -> Ready -> Streaming -> Stopping -> Disconnected
```

Receiver states:

```text
Listening -> Connected -> Handshake -> Ready -> Streaming -> Stopping -> Listening
```

Invalid packet order is handled as a protocol error:

- Before `HELLO`: reject non-`HELLO` packets.
- Before `STREAM_READY`: reject `AUDIO` and `STOP_STREAM` is allowed as graceful close.
- After `STOP_STREAM`: stop reading and release session ownership.

## Error Handling

Timeout values follow `docs/03-Protocol.md` defaults:

| Event | Timeout |
|------|---------|
| `HELLO` wait | 5 seconds |
| `WELCOME` wait | 5 seconds |
| `STREAM_READY` wait | 5 seconds |
| `PONG` wait | 15 seconds |
| TCP connect | 10 seconds |

Phase 1-A keeps timeout configuration local to the network classes. A persistent config file is not added.

Error handling rules:

- Unsupported protocol major returns `WELCOME(Result = Unsupported Protocol)` or closes the connection.
- Unsupported codec returns `ERROR(Unsupported Codec)` or `STREAM_READY(Result != Success)`.
- Malformed packet returns `ERROR(Invalid Packet)` when possible, then closes.
- Receiver busy returns `WELCOME(Result = Receiver Busy)`, then closes the new connection.
- Any unexpected socket failure releases session ownership.

## Testing Design

Tests are added at the smallest runnable boundary:

1. Protocol serialization tests compare exact bytes for header, strings, `HELLO`, `WELCOME`, `START_STREAM`, `PING`, and `STOP_STREAM`.
2. Parser tests cover the new packet types and invalid payload lengths.
3. Session tests verify valid order and invalid order transitions.
4. Receiver tests cover idle accept, busy reject, malformed packet close, and `PING -> PONG` echo.
5. Sender tests cover successful handshake, timeout, busy result, and protocol rejection.
6. One loopback integration test proves a fake sender and fake receiver can complete the Phase 1-A sequence.

Local environment note:

- This repository is currently on an `aarch64` Linux host.
- Android Gradle resource processing uses an `x86-64` `aapt2` binary, so full Android Gradle verification is expected to run on an `x86-64` runner or developer machine.
- Windows `.NET Framework 4.8` verification is expected to run on Windows.
- Local Linux remains useful for docs checks, golden packet checks, and any pure Python protocol fixture checks.

## Acceptance Criteria

Phase 1-A is complete when:

- Both implementations can serialize and parse all Phase 1-A packet types.
- Windows receiver can accept one sender and reject a second sender as busy.
- Android sender can complete the handshake against a compatible receiver on a supported build host.
- `PING` receives an identical `PONG` payload.
- `STOP_STREAM` releases the receiver session.
- Existing Phase 0 docs and golden packet checks still pass.
