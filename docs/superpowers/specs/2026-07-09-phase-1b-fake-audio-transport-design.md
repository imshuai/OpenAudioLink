# Phase 1-B Fake Audio Transport Design

Status: Ready for user review
Date: 2026-07-09

## Goal

Phase 1-B proves that Version 1 `AUDIO` packets can travel over the existing Phase 1-A TCP session before adding real Android capture, AAC encoding, Windows decoding, or playback.

The deliverable is a small fake audio transport path:

```text
TCP connect
HELLO -> WELCOME
START_STREAM -> STREAM_READY
AUDIO(fake AAC frame) x N
PING -> PONG
STOP_STREAM -> close
```

## Scope

Included:

- Shared `AUDIO` payload builder helpers for the existing Version 1 audio payload shape.
- Android fake audio sender path that sends a fixed number of deterministic `AUDIO` packets after `STREAM_READY`.
- Windows receiver session validation for incoming `AUDIO` packets while streaming.
- Minimal receiver-side observability for tests: received audio frame count and last valid audio payload.
- Unit tests for `AUDIO` payload bytes, valid audio acceptance, invalid audio rejection, and packet order.
- One loopback integration path using fake peers where the local platform can run it.

Excluded:

- Android `MediaProjection` permission flow.
- Android `AudioPlaybackCapture` and `AudioRecord`.
- Android `MediaCodec` AAC encoding.
- Windows Media Foundation AAC decoding.
- PCM playback.
- mDNS discovery.
- Foreground service.
- Full UI changes.

## Existing Baseline

Phase 1-A already provides:

- Cross-platform packet framing and stream readers/writers.
- `HELLO -> WELCOME` and `START_STREAM -> STREAM_READY` handshake.
- `PING -> PONG` echo and `STOP_STREAM` close.
- `AUDIO` packet type constants.
- `AudioPayloadValidator` on Android and Windows.
- `testdata/protocol/valid-audio-aac.bin` generated by `tools/protocol/generate_golden_packets.py`.

Phase 1-B extends this baseline instead of replacing it.

## Protocol Decisions

Phase 1-B follows `docs/03-Protocol.md` exactly for packet framing and `AUDIO` payload layout.

`AUDIO` packet type:

| Type | Name |
|-----:|------|
| `0x05` | `AUDIO` |

`AUDIO` payload:

```text
Codec              UInt8
Frame Number       UInt32
Capture Timestamp  UInt64
Frame Duration     UInt16
Encoded Size       UInt32
Encoded Data       Byte[Encoded Size]
```

The canonical fake Version 1 frame is the existing golden fixture payload:

```text
Codec = AAC-LC (1)
Frame Number = 1
Capture Timestamp = 123456789
Frame Duration = 20
Encoded Size = 4
Encoded Data = 0x11 0x22 0x33 0x44
```

The full canonical packet remains `testdata/protocol/valid-audio-aac.bin`:

```text
Packet Type = AUDIO (0x05)
Sequence Number = 2
Timestamp = 123456789
Payload = canonical fake frame above
```

Multiple fake frames increment only the frame number and packet sequence number. The encoded data may remain the same because Phase 1-B validates transport and framing, not audio content quality.

## Component Design

### Shared Protocol Helpers

Android and Windows add a small `AUDIO` payload builder using the existing packet writer primitives.

The helper builds only the Version 1 payload shape:

```text
Audio(codec, frameNumber, captureTimestamp, frameDuration, encodedData)
```

The existing `AudioPayloadValidator` remains the validation boundary. It checks:

- payload length is at least `AudioPayloadHeaderSize`;
- codec is AAC-LC for Version 1;
- encoded size matches the remaining payload bytes.

No shared cross-language generator is added. The Python golden packet generator remains the common binary source of truth.

### Android Sender

The Android sender keeps the Phase 1-A handshake sequence and inserts fake audio after `STREAM_READY` succeeds:

```text
send HELLO
wait WELCOME success
send START_STREAM
wait STREAM_READY success
send AUDIO fake frame 1
send AUDIO fake frame 2 ... optional fixed count
send PING
wait PONG
send STOP_STREAM
```

The fake audio count is local to `HandshakeClient` for Phase 1-B. A persistent configuration file is not added.

Recommended default for tests and smoke runs:

```text
FakeAudioFrameCount = 1
```

The fake sender does not try to pace frames in real time. Phase 1-B is a transport proof, not a latency or playback proof.

### Windows Receiver

The Windows receiver remains a single active session receiver.

While streaming:

- `AUDIO` packets are accepted.
- The payload is validated with `AudioPayloadValidator.ValidateAacPayload`.
- The session increments `AudioFramesReceived`.
- The session stores `LastAudioPayload` for tests.
- No response packet is sent for `AUDIO`.
- Invalid `AUDIO` payloads raise `PacketParseException`; the TCP receiver closes the connection and releases session ownership.

The receiver still responds to `PING` with byte-identical `PONG` payloads and stops on `STOP_STREAM`.

## State Machines

Android sender state adds one optional fake audio step after ready:

```text
Idle -> Connecting -> Handshake -> Ready -> SendingFakeAudio -> Streaming -> Stopping -> Disconnected
```

Windows receiver state remains the Phase 1-A state machine:

```text
Listening -> Connected -> Handshake -> Ready -> Streaming -> Stopping -> Listening
```

In `Streaming`, valid packet types are:

- `AUDIO`
- `PING`
- `STOP_STREAM`

## Error Handling

Error handling stays deliberately small:

- `AUDIO` before `STREAM_READY` is a protocol error.
- Invalid `AUDIO` payload is a malformed packet error and closes the session.
- Unsupported codec inside `AUDIO` is invalid for Version 1 and closes the session.
- Socket failures still release session ownership.
- Fake sender returns `false` for expected network or protocol failures.

No retry policy is added in Phase 1-B.

## Testing Design

Tests are added at the smallest runnable boundary:

1. Protocol serialization tests compare exact bytes for the fake `AUDIO` payload and `valid-audio-aac.bin`.
2. Parser/validator tests cover valid fake `AUDIO`, too-short payload, unsupported codec, and encoded-size mismatch.
3. Windows session tests verify:
   - `AUDIO` is rejected before `STREAM_READY`.
   - valid `AUDIO` increments frame count while streaming.
   - valid `AUDIO` produces no response packet.
   - invalid `AUDIO` throws `PacketParseException`.
4. Windows TCP loopback tests verify a fake peer can complete handshake, send one `AUDIO`, receive `PONG`, and stop.
5. Android sender tests verify successful run writes `HELLO`, `START_STREAM`, `AUDIO`, `PING`, and `STOP_STREAM` in order.
6. Existing Phase 0 and Phase 1-A golden checks must still pass.

Local environment note:

- Linux local checks remain useful for docs and protocol golden fixtures.
- Android full Gradle verification should run on an `x86-64` Android build host.
- Windows `.NET Framework 4.8` verification should run on Windows.

## Acceptance Criteria

Phase 1-B is complete when:

- Android can serialize deterministic fake `AUDIO` packets after `STREAM_READY`.
- Windows can validate and record valid `AUDIO` packets while streaming.
- Windows rejects `AUDIO` before streaming.
- Windows rejects malformed `AUDIO` payloads.
- `PING -> PONG` and `STOP_STREAM` still work after fake audio packets.
- Shared golden packet checks still pass.
- No real capture, encoding, decoding, playback, discovery, service, or UI scope is introduced.
